# Migration Guide `0.12.x -> 0.13.x`

This document outlines the necessary changes for migrating your tractusx-edc installation from versions 0.12.x to 0.13.x.
It also outlines some points that adopters and operators should pay close attention to when migrating from one version
to another.

This document is not a comprehensive feature list.

## 1. Federate Catalog removal

The Federate Catalog feature has not been proven to be usable, that's why it will be removed. The deletion of the table 
`edc_federated_catalog` will be done automatically by migration script. If you need to keep the data, please make sure 
that `tx.edc.postgresql.migration.federatedcatalog.enabled` is set to `false` before the migration.

## 2. Deprecated instances

The API `/business-partner-groups` has been removed, please use `/v3/business-partner-groups` instead.
Also the API `/v2/edrs` has been removed, please use `/v3/edrs` instead.
Additionally, several configuration parameters deprecated since version 0.8.x have also been removed.
Please use the updated configuration parameters instead.

[SqlBusinessPartnerGroupStoreExtension](../../edc-extensions/bpn-validation/business-partner-store-sql/src/main/java/org/eclipse/tractusx/edc/validation/businesspartner/store/SqlBusinessPartnerGroupStoreExtension.java):

| Old value                 | New value                      | New environment variable       |
|---------------------------|--------------------------------|--------------------------------|
| `edc.datasource.bpn.name` | `edc.sql.store.bpn.datasource` | `EDC_SQL_STORE_BPN_DATASOURCE` |

[SqlEdrLockExtension](../../edc-extensions/edr/edr-index-lock-sql/src/main/java/org/eclipse/tractusx/edc/edr/index/sql/lock/SqlEdrLockExtension.java):

| Old value                 | New value                      | New environment variable       |
|---------------------------|--------------------------------|--------------------------------|
| `edc.datasource.edr.name` | `edc.sql.store.edr.datasource` | `EDC_SQL_STORE_EDR_DATASOURCE` |

## 3. DID Document Service client type selection

**Breaking change:** A new property `tx.edc.did.service.client.type` is now required to select which
DID document service backend is used. Without this property, neither the DIV nor the IdentityHub
client will register, and DID service self-registration will silently skip.

### For existing DIV users

Add the following property to your configuration:

```properties
tx.edc.did.service.client.type=div
```

Or in the Helm chart `values.yaml`:

```yaml
iatp:
  didService:
    clientType: "div"
```

### For new IdentityHub users

Set the client type and provide the IdentityHub connection properties:

```properties
tx.edc.did.service.client.type=identityhub
tx.edc.ih.identity.api.url=https://<identityhub-host>/api/identity
tx.edc.ih.identity.api.key.alias=<vault-alias-for-api-key>
tx.edc.ih.participant.context.id=<participant-context-id>
```

Or in the Helm chart `values.yaml`:

```yaml
iatp:
  didService:
    clientType: "identityhub"
    identityHub:
      apiUrl: "https://<identityhub-host>/api/identity"
      apiKeyAlias: "<vault-alias>"
      participantContextId: "<context-id>"
```
